Understanding the Branch System

Before touching any command, understand this diagram. It is the foundation of everything.

main ← PRODUCTION. What the client/world sees. NEVER touch directly. ↑ └─ PR only (when ready to release) develop ← YOUR DAILY WORKING BRANCH. All work happens here or below. ↑ ├── feature/fee-management ← new feature branch ├── feature/student-demographics ← another feature branch ├── fix/attendance-500-error ← bug fix branch └── hotfix/login-crash ← urgent fix (branches from main)

What each branch means

Branch Purpose Who can push directly? How it gets updated
main Production — what the client sees Nobody PR from develop only
develop Integration — tested, stable work Direct push for tiny fixes only PR from feature/* or fix/*
feature/* One branch per new feature Author — push freely Branch from develop, PR back to develop
fix/* Non-urgent bug fixes Author — push freely Branch from develop, PR back to develop
hotfix/* Urgent production fix only Author — push freely Branch from main, PR to both main and develop
✅ Simple rule to remember: develop is your workspace. main is the final product. Never confuse the two.

One-Time Setup — First Time on a New Machine

Do this only once — the first time you set up the project on any computer.

Step 2.1
Tell Git who you are
Git stamps your name on every commit. Set this first.
Terminal run once 
# Replace with your actual name and email
git config --global user.name "Sankalp Deshpande"
git config --global user.email "your@email.com"

# Verify it saved correctly
git config --global user.name
# Should print: Sankalp Deshpande
Step 2.2
Clone the repository to your machine
This downloads the full project from GitHub to your computer.
Terminal run once 
# Navigate to where you want the project folder
cd C:/Users/YourName

# Clone the repo (downloads everything from GitHub)
git clone https://github.com/BuildWithInfovion/academic-saas.git

# Go into the project folder
cd academic-saas
Step 2.3
Get the develop branch on your machine
After cloning, you only have main locally. Get develop too.
Terminal run once 
# Download all branches from GitHub
git fetch origin

# Switch to develop and start tracking it
git checkout develop

# Confirm you are on develop
git branch
# Output should show:  * develop
#                        main
ℹ️ You only do Section 2 once per machine. After this, you never clone again. Just open the folder and start working.

Your Daily Start Routine

Every single day before you write one line of code, run these commands. This takes 10 seconds and prevents 90% of all problems.

Daily Start Routine — Run Every Morning every day 
# 1. Go to the project folder
cd C:/Users/sanka/academic-saas

# 2. Switch to develop (in case you left on another branch)
git checkout develop

# 3. Download the latest changes from GitHub to your machine
#    (Important if your partner pushed something overnight)
git pull origin develop

# 4. Check there are no surprises
git status
# Should say: "nothing to commit, working tree clean"

# ✅ You are ready to start working
⚠️ Never skip the git pull. If your partner pushed changes and you didn't pull, your code will be out of date and you'll get conflicts later.

Working on a New Feature (Full Step by Step)

This is the main workflow you'll use 95% of the time. Follow it for every new feature you build.

🚀
Example: Building the Fee Management Page
Use this same pattern for any feature you build
Complete Feature Workflow step by step 
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PHASE 1: PREPARE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Step 1: Always start from the latest develop
git checkout develop
git pull origin develop

# Step 2: Create your feature branch
#   Rule: feature/describe-what-you-are-building
git checkout -b feature/fee-management

# Confirm you are on the new branch
git branch
# Output:  * feature/fee-management
#            develop
#            main

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PHASE 2: CODE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# ... Open VS Code and write your code ...
# ... Make changes to files ...
# ... Test it locally ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PHASE 3: SAVE YOUR WORK (commit)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Step 3: See what files you changed
git status
# Shows modified files in red (not saved to git yet)

# Step 4: Stage the files you want to save
# Option A: Stage one specific file
git add apps/frontend/app/dashboard/fees/page.tsx

# Option B: Stage ALL changed files at once
git add .

# Step 5: Check staged files (now green)
git status

# Step 6: Commit (save a snapshot with a message)
git commit -m "feat: rebuild fee management with Indian fee heads"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PHASE 4: PUSH TO GITHUB
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Step 7: Push your branch to GitHub
git push origin feature/fee-management

# First time pushing this branch, use -u flag
# (this links your local branch to GitHub)
git push -u origin feature/fee-management

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PHASE 5: MERGE INTO DEVELOP (via GitHub PR)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Step 8: Go to GitHub in your browser
#   → You'll see: "feature/fee-management had recent pushes"
#   → Click "Compare & pull request"
#   → Make sure: base = develop, compare = feature/fee-management
#   → Click "Create pull request"
#   → Click "Merge pull request"
#   → Click "Confirm merge"

# Step 9: Come back to terminal, clean up
git checkout develop
git pull origin develop
git branch -d feature/fee-management
# Deletes the local branch (it's merged, no longer needed)

# ✅ Done! Your feature is now in develop.
#    main is completely untouched.

What happens step by step — explained simply

CommandWhat it does in plain English
git checkout -b feature/nameCreates a new branch and switches to it. Like creating a copy of develop to work on safely.
git statusShows you which files you changed. Red = not saved yet. Green = ready to commit.
git add .Marks ALL changed files as "ready to save". Like selecting files to put in an envelope.
git commit -m "message"Takes a snapshot of your work with a label. Like sealing the envelope with a note.
git push origin branch-nameSends your commits from your machine to GitHub. Like posting the envelope.
git branch -d branch-nameDeletes the branch locally after merging. Safe to do — the code is already in develop.

Quick Fix Directly on Develop

For very small changes (a typo, a label fix, a color change) you don't need a feature branch. Commit directly to develop.

🔧
Example: Fix a typo in the student form
Small, low-risk changes only
Quick Fix on Develop
# Step 1: Make sure you are on develop with latest code
git checkout develop
git pull origin develop

# Step 2: Make your small change in VS Code
# ... fix the typo ...

# Step 3: Stage and commit
git add .
git commit -m "fix: correct typo in student admission form label"

# Step 4: Push directly to develop
git push origin develop

# ✅ Done. No PR needed for tiny fixes.
⚠️ Use this only for truly tiny changes. If you're touching logic, building a feature, or changing more than 2-3 lines — always use a feature branch (Section 4).

Releasing to Production (develop → main)

When you have tested everything on develop and it's ready for the client, you release it to main.

🚢
Release — Moving develop to main
Do this when you're confident the code is stable
Release Workflow
# Step 1: Make sure everything is pushed to develop
git checkout develop
git status
# Should say: nothing to commit, working tree clean
git push origin develop

# Step 2: Go to GitHub in browser
#   → Click "Pull requests"
#   → Click "New pull request"
#   → base: main  ←  compare: develop
#   → Title: "release: v1.x - describe what's in this release"
#   → Click "Create pull request"
#   → Review the files changed (check nothing unexpected)
#   → Click "Merge pull request"
#   → Click "Confirm merge"

# Step 3: Pull main locally to stay in sync
git checkout main
git pull origin main

# Step 4: Tag this release (optional but good practice)
git tag v1.0.0 -m "Release v1.0.0 — Initial platform launch"
git push origin v1.0.0

# Step 5: Go back to develop for next work
git checkout develop

# ✅ main is now the latest production version.

Emergency Hotfix on Production

Something broke on main (production) and needs an immediate fix — you can't wait for the normal develop cycle.

🚨 Only use hotfix/* when production is broken. For everything else, use the normal feature branch from develop.
🔥
Emergency: Login is broken on production
Hotfix branches from main, not develop
Hotfix Workflow
# Step 1: IMPORTANT — branch from main, not develop
git checkout main
git pull origin main
git checkout -b hotfix/login-crash

# Step 2: Fix the bug
# ... make the fix in VS Code ...

# Step 3: Commit the fix
git add .
git commit -m "hotfix: fix login crash on wrong institution code"

# Step 4: Push the hotfix branch
git push origin hotfix/login-crash

# Step 5: On GitHub — create TWO pull requests
#
#   PR 1: hotfix/login-crash → main     (fixes production now)
#   PR 2: hotfix/login-crash → develop  (keeps develop in sync)
#
# Merge BOTH PRs.

# Step 6: Clean up
git checkout develop
git pull origin develop
git branch -d hotfix/login-crash

# ✅ Production is fixed. develop is also updated.

All Commands Reference Card

Bookmark this section. Every command you'll ever need, explained in plain English.

🔍 Checking Status

CommandWhat it does
git statusShows which files changed. Red = unsaved, Green = ready to commit.
git branchShows all local branches. The one with * is your current branch.
git branch -aShows ALL branches — local AND GitHub (remote).
git log --onelineShows recent commits in a short one-line format.
git log --oneline -10Shows only the last 10 commits.
git diffShows exactly what lines you changed (before staging).

🔀 Switching Branches

CommandWhat it does
git checkout developSwitch to the develop branch.
git checkout mainSwitch to the main branch.
git checkout -b feature/nameCreate a NEW branch called feature/name and switch to it.
git checkout -b fix/nameCreate a NEW fix branch and switch to it.

💾 Saving Work (Stage & Commit)

CommandWhat it does
git add .Stage ALL changed files — ready to commit.
git add filename.tsxStage only ONE specific file.
git add apps/frontend/Stage all changes inside a folder.
git commit -m "message"Save a snapshot with a descriptive message.
git commit --amend -m "new message"Fix the message of your LAST commit (before pushing).

☁️ Pushing & Pulling (GitHub sync)

CommandWhat it does
git push origin developSend develop commits to GitHub.
git push origin feature/nameSend a feature branch to GitHub.
git push -u origin feature/namePush and link local branch to GitHub (use first time).
git pull origin developDownload latest develop from GitHub to your machine.
git fetch originDownload branch info from GitHub without merging.

🧹 Cleaning Up

CommandWhat it does
git branch -d feature/nameDelete a local branch (safe — only works if merged).
git branch -D feature/nameForce delete a branch (even if not merged — careful!).
git stashTemporarily save uncommitted changes and clean your working directory.
git stash popBring back stashed changes.

🔗 Connecting to GitHub

CommandWhat it does
git remote -vShows which GitHub repo you're connected to.
git clone URLDownload a repo from GitHub for the first time.

Commit Message Rules

A good commit message tells anyone (including future you) what changed and why. Always write it in the format below.

Format
type: short description of what you did

Types available:
feat     → New feature or page
fix      → Bug fix
hotfix   → Urgent production fix
refactor → Code cleanup (no new feature, no bug fix)
chore    → Config, setup, dependency update
docs     → Documentation only

Examples — Good vs Bad

❌ Bad (vague)✅ Good (descriptive)
fixed stufffix: student list crashes when page param is missing
updatefeat: add bulk promote/holdback for students
changesrefactor: move fee head seeding to institution setup wizard
wipfeat: receptionist portal — inquiry management page
asdfhotfix: login broken when institution code has uppercase letters

Golden Rules — Do & Don't

✅ Always Do
  • Pull before you start working each day
  • Create a feature branch for every new feature
  • Write clear commit messages
  • Test locally before pushing
  • Merge feature branches back to develop via PR
  • Keep branches short-lived (1-3 days max)
  • Delete branches after merging
  • Stay on develop as your base
❌ Never Do
  • Push directly to main
  • Code on main branch
  • Merge feature directly to main (bypass develop)
  • Commit .env files or passwords
  • Write vague commit messages like "changes"
  • Leave huge PRs with 50+ files (keep PRs small)
  • Force push to main or develop
  • Skip the daily git pull
