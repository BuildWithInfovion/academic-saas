Academic SaaS Platform
System Architecture & Complete Product Reference

Infovion Internal Documentation  ·  Confidential  ·  April 2026  ·  v2.0

Purpose: This is the authoritative internal reference for the Infovion Academic SaaS platform. It documents every layer of the product — features, architecture, data models, security, and scaling strategy — for technical leadership, product teams, and engineering onboarding.

01 Product Overview

Infovion is a multi-tenant school management SaaS platform. Each subscribing school (institution) receives a fully isolated data environment within a shared infrastructure. The product serves three distinct user populations over separate URL paths with independent authentication contexts:

Institution Operators

School admins who manage all institutional data: students, staff, fees, academics, timetables, and settings via /dashboard.

Portal Users

Role-specific stakeholders — teachers, parents, principals, accountants — consuming structured data through their own dedicated portal at /portal/[role].

Platform Admins (Infovion)

Infovion's own team who onboard clients, manage subscriptions, and handle support tickets via a private console at /platform.

02 Product Objectives

Operational Efficiency

Digitise all school operations — admissions, attendance, fees, exams, timetables — eliminating paper-based workflows entirely.

Parent Transparency

Real-time visibility into attendance, marks, fees, and school events through a mobile-friendly parent portal accessible from any browser.

Multi-Tenant Isolation

Hundreds of schools on a single deployment with zero cross-tenant data exposure, enforced at both the application and query layer.

Role-Gated Access

Every user sees exactly what their role permits — no over-exposure of data, no accidental cross-role visibility.

SaaS Manageability

Complete lifecycle control for Infovion: onboarding, subscription management, feature flags, and support from a private admin console.

Security-First Design

httpOnly cookies, token rotation, TOTP 2FA, rate limiting, audit logging, and tenant guards throughout the stack.

04 Complete Feature List — Operator Dashboard

Student Management

Student Admission
Full admission form: personal details, parent details, address, demographics (blood group, nationality, religion, caste category, Aadhaar), disability status, previous TC details. Admission number auto-assigned or manually specified.
Parent Account Auto-Provisioning
When a student is admitted with a parent phone number, a parent user account is automatically created. The phone becomes the login credential. No separate manual step required.
Student Directory
Searchable, filterable directory of all students. Filter by class, section, status (active/inactive). View full student profile.
Student Profile Edit
Update any student field post-admission. Soft delete (deactivate) without losing historical data including attendance, fees, and marks.
Sibling Linking
Link siblings via a shared siblingGroupId for batch fee operations and family-level reports.
Student Promotion
Bulk promote students from one academic unit to another (end-of-year class promotion). Source unit becomes empty; students are reassigned to the target unit.
Parent Password Reset
Operator can directly reset a parent's password and receive the new password in the response (displayed once, then discarded).

Academic Structure

Academic Year Management
Create and manage academic years (e.g. 2025-26). Mark one as current. Each year has a start and end date.
Academic Unit Hierarchy
Create grades (level 1) and sections within each grade (level 2). Self-referential parent-child structure. Unlimited depth supported by the data model.
Class Teacher Assignment
Assign a class teacher to each section. Teacher is linked via classTeacherUserId on AcademicUnit.
Subject Management
Create subjects for the institution. Assign subjects to academic units. Link a specific teacher to each subject within a unit.

Fee Management

Fee Head Configuration
Define fee categories: Tuition Fee, Transport, Lab Fee, Sports Fee, etc. Custom fee heads supported.
Fee Structure Setup
Assign fee amounts per academic unit + year + fee head combination. Supports installment breakdown with due dates per installment.
Payment Collection
Record payments per student: amount, fee head, payment mode (cash, cheque, online), receipt number. Receipt number is unique per institution and auto-incremented.
Student Balance Calculation
Real-time balance per student per academic year: total dues (from fee structure) minus total payments made.
Defaulter Detection
List all students with outstanding fees, grouped by class. Filter by academic year. Used for fee reminder campaigns.

Examination System

Exam Creation & Scheduling
Create named exams (Mid-Term, Final, Unit Test). Add subject-level schedule per unit: exam date, time, max marks, passing marks, exam centre.
Mark Entry
Enter marks per student per subject. Support for marking students as absent (distinct from zero marks). Bulk entry per class.
Result Reports
Class-wise result summary: rank list, percentage, subject-wise pass/fail count. Student-level report card with all subjects.
Result Visibility Control
Exam status (draft / published / completed) controls when results are visible to parents and students.

Staff Management

Staff User Creation
Create staff accounts with email, phone, role assignment. Multiple roles per user supported via UserRole junction table.
Role Assignment
Assign any combination of roles to a staff member. Role permissions are JSON arrays configurable per institution.
Staff Activation / Deactivation
Deactivate a staff account without deleting it. Login is rejected for inactive accounts. Historical data retained.
Staff Password Management
Admin-initiated password reset returns the new password once. Staff can self-reset via email OTP flow.
Staff Attendance Management
View all staff attendance records. Admin can override any staff member's daily attendance. Approve or reject leave requests.

Administrative Tools

Transfer Certificate Workflow
Request TC → approve → issue. Captures immutable student snapshot at request time (academic data, fee status, attendance). Sequential TC number assigned on issue.
Inquiry Management
Track pre-admission inquiries through a status pipeline: new → contacted → visited → enrolled / dropped. Add notes at each stage.
Announcement Publishing
Post announcements targeted to specific roles (or all). Pin important notices. Set expiry date. Immediately visible to all targeted portal users.
Institution Profile
Manage institution name, address, contact, website, board type, logo. Feature flags configurable per institution.
Timetable Setup
Build weekly timetable per class: day × period grid. Assign subject and teacher per slot. Bulk save replaces the full unit timetable at once.
Calendar Management
Add institution calendar events: holidays, exam dates, parent meetings, vacations. Visible to all portal users.
TOTP 2FA Setup
Operators and staff can optionally enable two-factor auth via any standard authenticator app. 8 single-use backup codes generated on setup.

12 Authentication System Design

The auth system solves two production problems that emerged with multi-context usage:

Problem 1 — Session Collision

When a school admin and parent open the platform in separate tabs on the same browser, a shared cookie name caused one login to overwrite the other's session. Refreshing either tab produced a blank screen or wrong user context.

Problem 2 — Blank Screen on Refresh

Access tokens were stored in-memory only. Every page refresh required a network round-trip to restore the session, causing a 200–500ms blank screen flash before the UI could render.

Solution 1 — Separate Cookie Names

Each auth context gets its own httpOnly cookie: auth_rt_op for operators, auth_rt for portal users, platform_rt for Infovion admins. The browser enforces cookie namespacing — no collision possible.

Solution 2 — sessionStorage Persist

Zustand persist middleware caches the access token in sessionStorage per tab. On refresh, the token is restored synchronously before any network call. Post-hydration getState() read pattern ensures correct timing.

17 Database Schema

18 Multi-Tenancy Design

Pattern: Shared Database, Separate Rows. All institutions share the same PostgreSQL tables. Every institution-scoped row includes an institutionId foreign key. The application enforces isolation at two layers: (1) TenantGuard compares the JWT's institutionId with the request header, and (2) every Prisma query includes where: { institutionId }. A compromised token from Institution A cannot read Institution B's data.
Soft Deletes: Critical models (students, academic units, announcements, exams, fee structures, calendar events, TCs) use deletedAt: DateTime? rather than hard deletes. This preserves audit history and allows data recovery without DB restoration. All list queries filter where: { deletedAt: null }.

21 Security Architecture

Security is applied in multiple layers. Compromise of any single layer does not result in a breach — each layer provides independent protection.

Security Layers (outermost to innermost) [Layer 1] DNS / TLS HTTPS enforced on all domains. Cookies marked Secure=true. [Layer 2] Vercel Edge Middleware Cookie presence check before any page/compute runs. [Layer 3] CORS Only allowlisted origins accepted. Blocks cross-origin API calls. [Layer 4] Rate Limiting Per-IP and per-user limits on all login endpoints. [Layer 5] ValidationPipe Strips and rejects any field not declared in a DTO. [Layer 6] AuthGuard JWT signature + expiry validation on every authenticated call. [Layer 7] TenantGuard Cross-tenant request blocked even with a valid token. [Layer 8] RolesGuard Insufficient-permission requests rejected per endpoint. [Layer 9] Query-level isolation All DB queries include institutionId — data physically cannot leak.

22 Security Implementation Detail

Threat: Session Collision Between User Types
Implementation: Three distinct httpOnly cookie names (auth_rt_op, auth_rt, platform_rt) set by the backend at login based on role detection. Refresh endpoints only read their own cookie — /auth/refresh-op reads auth_rt_op, ignores auth_rt. Frontend Zustand stores use different sessionStorage keys per context. Browser enforces complete namespace separation.
Threat: Cross-Tenant Data Access
Implementation: TenantGuard compares req.user.institutionId (from JWT, set at login, cannot be modified by client) against the X-Institution-ID header sent by the frontend. If they differ, request is rejected with 403 before any service method runs. Additionally, every Prisma query includes where: { institutionId } — a compromised token still cannot read another institution's rows because the SQL condition filters them out.
Threat: Refresh Token Replay Attack
Implementation: Refresh tokens are never stored in plaintext. The backend bcrypt-hashes the token and stores the hash in the RefreshToken table with isRevoked: false. On refresh: compare submitted token against hashes for the user, verify not revoked and not expired. On success: mark old token as isRevoked: true, generate new token pair. Even if an attacker intercepts a refresh token in transit, using it once invalidates it — the next legitimate use would be rejected.
Threat: XSS Token Theft
Implementation: Refresh tokens are in httpOnly cookies — completely inaccessible to JavaScript. Access tokens are stored in sessionStorage (not localStorage) and have a 15-minute lifetime. If XSS extracts an access token, it expires in at most 15 minutes. sessionStorage is also cleared when the tab closes, further limiting exposure window.
Threat: Brute Force Login
Implementation: Two independent rate limiters: LoginRateLimitGuard for /auth/login and ParentLoginRateLimitGuard for /auth/parent/login. Each tracks attempts per IP (max 10 per window) and per user (max 8 per window). Returns HTTP 429 with Retry-After header. Separate guards prevent one endpoint's traffic from consuming the other's limit pool.
Threat: SQL Injection
Implementation: All database access goes through Prisma's parameterised query builder. Zero raw SQL strings that include user input anywhere in the codebase. The only raw SQL is SELECT 1 in the health check — no user data involved. Prisma's generated client types prevent passing arbitrary strings where typed values are expected.
Threat: Input Injection / Mass Assignment
Implementation: Global ValidationPipe with whitelist: true strips any fields not declared in the DTO, and forbidNonWhitelisted: true returns 400 if unexpected fields are present. Every endpoint has a typed DTO with class-validator decorators. Untrusted user input cannot reach the service layer in any unexpected shape.
Threat: CSRF
Implementation: All state-changing API calls require a valid Authorization: Bearer <token> header. CSRF attacks (which forge same-origin cookie submission) cannot forge this header. Additionally, cookies use sameSite: 'lax' which prevents them from being sent on cross-site POST requests initiated from third-party sites.
Threat: Password Reset Enumeration
Implementation: POST /auth/forgot-password always returns the same constant success response regardless of whether the email exists in the system. An attacker cannot determine which emails have accounts by observing response differences.
Threat: Platform Admin Account Takeover
Implementation: PlatformAdmin has failedLoginCount, lockedUntil, and lastFailedAt fields. After exceeding the threshold, login is blocked until lockedUntil time passes. activeSessionId enforces single-session: a new successful login updates this field, immediately invalidating any previous session — even if the previous session's cookie is still valid.
Threat: Data Exposure via Error Messages
Implementation: PrismaExceptionFilter catches all Prisma-specific error codes (P2002 unique violation, P2025 record not found, etc.) and maps them to clean HTTP responses with generic messages. Raw DB error messages, table names, and column names never reach the client response.
Threat: Audit Trail Gaps
Implementation: Global AuditLogInterceptor fires on every non-GET response automatically — no per-route configuration needed. Records: action name, entity type, entity ID, before-value (JSON), after-value (JSON), IP address, user ID, and timestamp. The interceptor operates at the NestJS interceptor level, so it cannot be bypassed by controller code.
Threat: Support Ticket Spoofing
Implementation: POST /support/ticket extracts institutionId, institutionName, and submittedBy (email/phone) from the validated JWT — not from the request body. Users cannot claim to be from a different institution or submit under a different identity. The DTO only accepts subject and message from user input.

25 Scale Tiers — Schools & Migration Roadmap

TIER 1 1 – 50 Schools Current Architecture

Infrastructure: Current setup — single Railway backend, Railway managed PostgreSQL, Vercel frontend. No changes required.

Estimated data volume: ~50 schools × 500 avg students = 25,000 students. Fee payments: ~500,000 records/year. Attendance: ~5M records/year. All fits comfortably on a single PG instance.

Action required: None beyond monitoring. Set up Railway and Vercel alerts for response time and DB CPU.

  • Keep the in-memory HTTP cache for list endpoints — eliminates repeat DB reads for static data (academic units, subjects, roles)
  • Run EXPLAIN ANALYZE on the 5 most-called queries monthly to catch index drift
  • Ensure Railway PostgreSQL is on a plan with automated daily backups
TIER 2 50 – 200 Schools Optimise & Observe

Infrastructure changes: Upgrade Railway PostgreSQL to a larger plan. Add a Redis instance for shared rate limiting and caching. Upgrade backend to a higher-spec Railway instance.

Estimated data volume: 200 schools × 800 avg students = 160,000 students. Fee payments: ~4M records/year. Attendance: ~40M records/year. DB size: ~50–100 GB.

Specific actions:

  • Redis rate limiting: Replace in-process rate limit counters with Redis-backed @nestjs/throttler store. Required before horizontal backend scaling — in-memory counters are per-process and would allow limit bypass across instances.
  • Redis caching: Replace AppCacheModule (in-memory) with @nestjs/cache-manager Redis adapter. Shared cache survives process restarts and works across multiple backend instances.
  • Database read replica: Migrate reporting queries (exam result summaries, attendance reports, fee defaulter lists) to a read replica. Prisma supports read/write splitting with $extends.
  • Audit log partitioning: Partition the audit_logs table by institutionId range or month to prevent unbounded table growth from degrading query performance.
  • Refresh token cleanup job: Schedule a nightly cron that deletes RefreshToken rows where expiresAt < NOW() OR isRevoked = true. Without this, the table grows indefinitely.
TIER 3 200 – 1,000 Schools Horizontal Scale

Infrastructure changes: Multiple backend instances behind a load balancer. Dedicated Redis cluster. PostgreSQL with read replica. Consider migrating to AWS RDS or Neon for better connection pooling (PgBouncer).

Estimated data volume: 1,000 schools × 1,000 avg students = 1M students. Attendance records: ~200M/year. Total DB size: ~500 GB – 1 TB. This is where single-DB limits become real.

Specific actions:

  • Horizontal backend scaling: NestJS is already stateless — JWT validation is pure CPU with no shared process state. Deploy 3–5 instances behind a Railway load balancer or migrate to a container orchestration platform (ECS, Cloud Run).
  • Connection pooling: At this scale, each backend instance opening its own PG connection pool hits the database's max_connections limit quickly. Add PgBouncer between the backends and PostgreSQL. Target: transaction-mode pooling for short-lived queries.
  • Database sharding strategy: Consider migrating from shared-DB to a schema-per-institution or database-per-institution-group model. Prisma supports multi-database setups. Group institutions by region or signup date. This is a major migration — plan it with zero-downtime blue/green cutover.
  • Async processing: Heavy operations (bulk mark entry for large exams, attendance report generation for the full year) should be moved to a background job queue (Bull + Redis). Return a job ID immediately; client polls for completion. This prevents HTTP timeout on large datasets.
  • CDN for static assets: Institution logos and any uploaded files should go to S3 + CloudFront rather than being served through the NestJS process. Current setup does not handle file uploads — implement this before allowing photo uploads at scale.
  • Observability: Integrate OpenTelemetry for distributed tracing. At this scale, a slow query or a misbehaving interceptor can degrade the entire cluster silently. Sentry for error tracking, Datadog or Grafana for metrics.
TIER 4 1,000+ Schools Distributed Architecture

This tier requires architectural decisions that should be planned at Tier 3, not when you arrive at Tier 4.

Estimated data volume: 1,000+ schools × variable students = multi-million student records. DB size: multi-TB. At this scale, a single PostgreSQL instance — even on the highest hardware — becomes a coordination bottleneck.

Specific actions:

  • Tenant database isolation: Each institution (or group of 50–100 institutions) gets its own PostgreSQL database. A routing layer (tenant resolver) maps institutionId → database connection string. Prisma's multi-client pattern supports this. Benefits: complete data isolation, independent backup/restore, no cross-tenant query interference.
  • Microservice decomposition: Split the NestJS monolith into bounded services: Auth Service, Student Service, Fee Service, Exam Service. Communicate via a message broker (RabbitMQ or Kafka). Each service owns its data store. The monorepo structure already separates modules cleanly — this decomposition follows existing module boundaries.
  • Event-driven architecture: Parent portal reads (attendance, marks, fees) generate high read volume. Denormalise into a read-optimised store (e.g. Elasticsearch or a separate PG read database) that is updated by events when writes happen. Parent portal queries the fast read store; admin writes go to the authoritative DB.
  • Geographic distribution: Deploy backend and database clusters per region (India: Mumbai, Singapore). Use a global load balancer to route institutions to the nearest cluster. Reduces latency significantly for schools in different geographic zones.
  • Background job platform: Migrate from Bull (Redis-backed, single cluster) to a distributed job platform (Temporal.io or AWS SQS + Lambda workers). Handle bulk operations, report generation, and scheduled tasks at scale without timeouts.
  • Dedicated analytics pipeline: Offload all reporting queries to a separate analytics database (ClickHouse or BigQuery) populated by a change data capture (CDC) stream from the primary DB. Reporting never hits the transactional database.

27 Migration Readiness — What Exists Today

The current architecture already makes several correct decisions that simplify future scaling:

Stateless Backend

JWT validation is pure CPU. No session state in memory. Adding more backend instances requires zero coordination. Load balancing is trivial — any instance can handle any request.

Abstracted Cache Layer

AppCacheService is already an abstraction over the cache manager. Swapping from in-memory to Redis requires changing one module import, not refactoring service code.

institutionId on Every Row

The shared-DB multi-tenant design means moving to per-institution databases is a data migration challenge, not a code challenge. All queries already include institutionId — no query refactoring needed.

Module-Per-Feature Backend

NestJS modules map directly to potential microservices (AuthModule, FeesModule, ExamModule). If decomposition is needed, each module already owns its service, controller, and DTOs.

Prisma Migrations

All schema changes are version-controlled migrations that apply atomically. Zero-downtime schema changes are achievable by adding nullable columns first, backfilling, then adding constraints.

Feature Flags Ready

Institution.features is a JSON column. Per-institution feature gating can be added without schema changes — just add a feature key and check it in the relevant service.
