Project Overview

This is a multi-tenant academic management SaaS platform built for Infovion Technologies. It manages the full lifecycle of a school: admissions, attendance, examinations, fees, staff, and communication — across multiple schools (institutions) from a single deployment.

The first client is a K-12 school in India. The platform is designed to scale to colleges and other institution types.

🏫

Multi-Tenant

Multiple schools share one database and backend. Data is isolated by institution_id on every table.

👥

7 Roles

Director, Operator, Principal, Teacher, Student, Parent, Receptionist — each with its own portal and permissions.

🇮🇳

India-Optimised

Fields like Aadhar, caste category, board type (CBSE/ICSE/State), and Indian fee heads are built-in.

⚡

Monorepo

Backend and Frontend live in the same Git repository under apps/backend and apps/frontend.

Soft Delete

Three models — User, Student, Institution — use soft delete. The PrismaService globally intercepts Prisma operations:

// prisma.service.ts — what happens automatically

// .delete() is converted to:
.update({ data: { deletedAt: new Date() } })

// .findMany() / .findFirst() automatically adds:
.where({ ...originalWhere, deletedAt: null })

This means deleted users and students disappear from all queries without any extra code in individual services.

API Client — lib/api.ts

All API calls from the frontend go through the apiFetch() wrapper:

// Automatically adds auth headers and handles token refresh
const data = await apiFetch('/students?unitId=abc123');
const result = await apiFetch('/attendance', {
  method: 'POST',
  body: { academicUnitId, date, records: [...] }
});

What it handles:

  • Reads accessToken and institutionId from Zustand store
  • Adds Authorization: Bearer <token> and X-Institution-ID headers
  • On 401: silently calls /auth/refresh, retries the original request once
  • If refresh also fails: calls logout() and redirects to login
  • Returns parsed JSON or throws a human-readable error string

Token Lifecycle

  • Access Token (JWT): 15 minutes. Signed with JWT_SECRET (128-char hex). Payload includes sub (userId), institutionId, roles[], permissions[].
  • Refresh Token: 7 days. Stored as a hash in the database (RefreshToken table), not the raw token. On use, old token is revoked and a new one is issued (rotation).
  • Logout: All refresh tokens for the user are revoked in the database.

JWT Strategy — src/modules/auth/jwt.strategy.ts

After the JWT is decoded, the strategy performs a live database check:

  • Finds the user by sub (user ID)
  • Verifies isActive === true and deletedAt === null
  • If either check fails, the request is rejected with 401
  • This means deactivating a user in the DB immediately invalidates their JWT — no need to wait for expiry
Environment variable requirement: JWT_SECRET must be set in apps/backend/.env. The app uses configService.getOrThrow('JWT_SECRET') — it will crash on startup if this variable is missing. There is no fallback value.

Multi-Tenancy

Multiple schools share one database and one backend deployment. Data isolation is handled at the application layer, not at the database level (DB-level RLS is planned for a future phase).

How Isolation Works

  1. Every tenant table has an institutionId column.
  2. Every API call (except /auth/login and /platform/*) must include an X-Institution-ID header.
  3. TenantMiddleware reads this header and attaches it to the request object.
  4. Every service method includes .where({ institutionId }) in all Prisma queries.
// Example in students.service.ts:
findAll(institutionId: string) {
  return this.prisma.student.findMany({
    where: { institutionId, deletedAt: null } // ← always scoped
  });
}

How TenantMiddleware Works

// tenant.middleware.ts
// Applied to all routes EXCEPT /auth/* and /platform/*

use(req, res, next) {
  const institutionId = req.headers['x-institution-id'];
  if (!institutionId) throw new BadRequestException('X-Institution-ID header required');
  req.tenant = { institutionId };
  next();
}
No DB-level RLS yet. A bug in application code that forgets to add where: { institutionId } would expose cross-tenant data. Adding PostgreSQL Row-Level Security (RLS) is listed as a P2 task.

🔒

Security

Security baseline is solid. JWT, bcrypt, no secrets in code, CORS via env, global error handling, soft deletes, audit trail, and sessionStorage token isolation are all in place.
🔑

JWT_SECRET

128-character hex string stored only in apps/backend/.env. The app crashes on startup if this is missing — no fallback.

🔐

Password Hashing

bcrypt with salt rounds. Plaintext passwords are never stored or logged.

🔄

Refresh Token Rotation

Refresh tokens are hashed in the DB. Each use issues a new token and revokes the old one. Logout revokes all tokens.

🌐

CORS

Controlled by the CORS_ORIGIN environment variable (comma-separated list of allowed origins). Wildcard origin: true was removed.

🏢

Tenant Isolation

Every protected request requires X-Institution-ID header. Every DB query scoped by institutionId.

🗑️

Soft Delete

Users and students are never physically deleted. deletedAt is set instead. JwtStrategy checks deletedAt: null — deleted users can't authenticate.

📝

Audit Logging

Global AuditLogInterceptor records every API action: who, when, what, old value, new value. Cannot be bypassed at the route level.

✅

Input Validation

Global ValidationPipe with whitelist: true and forbidNonWhitelisted: true — unknown fields in request bodies are stripped and rejected.

🗂️

Tab Isolation

Auth tokens stored in sessionStorage (not localStorage). Each browser tab has its own session — avoids multi-tab token collision.

Adding a New Feature Module (Backend)

  1. Create folder: src/modules/myfeature/
  2. Create myfeature.module.ts, myfeature.controller.ts, myfeature.service.ts
  3. All service methods must accept and use institutionId
  4. All Prisma queries must include where: { institutionId }
  5. Import MyFeatureModule in app.module.ts
  6. Add @UseGuards(AuthGuard, RolesGuard) and @Permissions('feature.read') to routes

Adding a New Frontend Page

  1. Create file in the appropriate portal folder: app/portal/teacher/mypage/page.tsx
  2. Add "use client" at top if the page has any state or effects
  3. Use apiFetch() from lib/api.ts for all API calls — never use fetch() directly
  4. Add the page to the portal's menuItems in its layout.tsx
  5. PortalShell handles auth and role guards automatically

Database Changes

  1. Edit apps/backend/prisma/schema.prisma
  2. Run npx prisma migrate dev --name describe_change
  3. Run npx prisma generate
  4. On Windows, if Prisma DLL lock occurs: taskkill /F /IM node.exe → npx prisma generate → restart servers
