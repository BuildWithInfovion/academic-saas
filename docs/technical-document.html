Executive Summary

What the platform is, who it serves, and why it was built this way.

The Infovion Academic SaaS Platform is a full-stack, multi-tenant school management system purpose-built for the Indian K-12 education market. It delivers a unified digital backbone for every operational workflow of a school — from student admission and daily attendance through fee collection, examination management, timetable scheduling, staff payroll, and government-format Transfer Certificate generation.

The platform operates as a Software-as-a-Service model: a single production deployment serves any number of independent institutions, each fully isolated at the data layer with its own users, roles, fee plans, academic calendar, branding, and compliance records. Institutions access the platform at app.buildwithinfovion.com with the API served from api.buildwithinfovion.com.

🎯
Primary Market
Indian private and government-aided schools (K–12) requiring digital school management with India-specific compliance fields (UDISE code, GSTIN, caste categories), IST-aware fee and attendance logic, and government-format Transfer Certificate generation.

Key Differentiators

🔐 Enterprise-Grade Security

  • Custom RFC 6238 TOTP 2FA — zero extra dependency, inline implementation using Node.js crypto
  • Per-request audit log with automatic sensitive-field redaction via setImmediate()
  • HttpOnly cookies — refresh tokens never accessible to JavaScript
  • Rate limiting on all three authentication surfaces (staff, parent, platform admin)
  • Single active session enforcement for platform admins with per-IP login audit trail

🏗 True Multi-Tenancy

  • Shared database, row-level isolation — every table carries institutionId
  • Five independent isolation layers: middleware → guard → JWT binding → service → DB FK
  • JWT institutionId cross-validated against X-Institution-ID request header
  • Plan-based JSON feature flags control which modules are available per institution
  • Platform admin layer entirely separate from all institution-level users

🇮🇳 India-Specific Compliance

  • UDISE code, GSTIN, PAN, recognition number on institution profile
  • Caste categories: General, OBC, SC, ST — plus EWS and disability flags on student records
  • Government-format Transfer Certificate with immutable snapshot and conduct grade
  • IST-aware date arithmetic throughout (IST_MS = 5.5 × 60 × 60 × 1000)
  • Medium of instruction, school type, management type fields

⚡ Modern Production Stack

  • NestJS 11 + TypeScript 5.7 strict mode — no implicit any anywhere
  • Next.js 16 App Router + React 19 with edge middleware for route protection
  • Prisma 5 ORM — all DB queries fully type-safe, schema-first
  • Containerized Docker deploy on Render Singapore — closest region to India
  • PgBouncer transaction-mode connection pooling for runtime query efficiency

Multi-Tenancy Architecture

Shared database, row-level isolation. A single PostgreSQL cluster hosts all institutions. Five independent mechanisms enforce data boundaries.

🏗
Design Decision: Shared Schema
Migrations apply once, monitoring is unified, and tenant provisioning is instant — one INSERT into the Institution table. The trade-off of application-layer isolation (vs. schema-per-tenant) is addressed through five independent enforcement layers described below. At current scale, this is the optimal approach.

Five Isolation Layers

LayerMechanismWhat It Prevents
L1 — TransportTenantMiddleware reads X-Institution-ID header on every non-exempt request. Throws 400 Bad Request if absent.Requests reaching the service layer without a declared institution context.
L2 — Route GuardTenantGuard queries Institution table, verifies status = ACTIVE. Result cached 5 min per institutionId via AppCacheModule.Access to suspended or deleted institution data. DoS via repeated institution lookup.
L3 — JWT BindingAuthGuard validates that JWT.institutionId === X-Institution-ID header value on every request.Token reuse across institutions. An operator of School A cannot use their token against School B's API.
L4 — Service LayerEvery service method receives institutionId from the @Tenant() decorator and passes it to every single Prisma query as a where clause filter.Cross-tenant data leakage even if layers 1–3 were bypassed.
L5 — Database FKAll tenant-scoped tables have institutionId String with a FK to Institution. Compound indexes on (institutionId, id) and (institutionId, field) for query performance.Orphaned records. Provides query-plan isolation via indexed scans per institution.

Institution Provisioning Flow

Step 1 — Onboard via Platform Admin
Infovion admin calls POST /platform/clients with institution name, unique code slug, plan code, and contact. A new row is inserted into Institution with status = ACTIVE. Takes < 1 second — no provisioning pipeline.
Step 2 — Director User Created
First operator user (Director role) is provisioned at the same time. They receive login credentials and have immediate dashboard access with full RBAC permissions.
Step 3 — Institution Self-Onboards
Director configures institution branding (logo, stamp, signature via Cloudinary signed uploads), compliance fields (UDISE, GSTIN, PAN), bank account details, and creates academic year + class hierarchy.
Step 4 — Staff & Students Onboarded
Operator creates teacher users with assigned RBAC roles. Students are imported via CSV bulk import (with rollback-by-batch-ID) or created individually. Parents are linked to student records. All data is immediately isolated to the institution's institutionId.

Edge Middleware Route Protection

Next.js middleware.ts runs at the edge before any page component renders. It checks the presence of the session cookie for each route prefix and redirects to the appropriate login page if absent — no client-side hydration occurs for unauthenticated users.

// middleware.ts — simplified logic
export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  if (pathname.startsWith('/dashboard')) {
    if (!request.cookies.get('auth_rt_op')) return redirect('/login');
  }
  if (pathname.startsWith('/portal')) {
    const cookie = request.cookies.get('auth_rt') || request.cookies.get('portal_ready');
    if (!cookie) return redirect('/login');
  }
  if (pathname.startsWith('/platform')) {
    if (!request.cookies.get('platform_rt')) return redirect('/platform/login');
  }
}

Zustand Auth Stores

Two completely separate Zustand stores — one for operator/dashboard users, one for portal users. A single browser session can hold both simultaneously, enabling a school administrator to have the dashboard open in one tab and a parent portal in another without any state collision.

Operator Auth Store

  • Stores access token, user object, institutionId
  • Persisted to sessionStorage (cleared on tab close)
  • Silent refresh calls POST /auth/refresh-op
  • Singleton isRefreshing guard prevents concurrent refresh storms

Portal Auth Store

  • Identical structure but entirely separate instance
  • Persisted independently — no interference with operator store
  • Silent refresh calls POST /auth/refresh
  • /portal/select-role page for users with multiple portal roles

API Client — lib/api.ts

FeatureImplementation Detail
Request deduplicationConcurrent GET requests to the same URL share one in-flight Promise. Only one network request is made regardless of how many callers are waiting.
Response caching5-minute TTL cache for GET responses on static endpoints (classes, subjects, academic years, institution profile). Cache keyed by full URL string.
Auto token refreshOn 401 Unauthorized, transparently calls the appropriate refresh endpoint (/auth/refresh or /auth/refresh-op) and retries the original request exactly once with the new token.
Refresh storm preventionA singleton isRefreshing boolean and a queue of pending resolvers ensure that if multiple requests hit 401 simultaneously, only one refresh call is made. All queued requests resolve after the single refresh completes.
Tenant header injectionAutomatically appends X-Institution-ID header from the active auth store on every non-platform request.
Cache invalidationOn any mutation (POST/PATCH/DELETE), all cached response entries whose keys begin with the same route prefix are evicted immediately.

TOTP Two-Factor Authentication (RFC 6238)

Zero-dependency inline implementation using only Node.js built-in crypto module. No external TOTP library is installed.

StepDetail
Secret generationtotpGenerateSecret() — 20 cryptographically random bytes → Base32 encoded using RFC 4648 alphabet (A-Z, 2-7)
OTP generationtotpGenerate(secret, step) — HMAC-SHA1 of 8-byte big-endian counter (Unix time ÷ 30); dynamic truncation per RFC 4226 §5.4; 6-digit output
VerificationtotpVerify(token, secret, window=1) — checks current step ±1 (±30 seconds clock drift tolerance)
Backup codes8 one-time codes generated at setup — SHA-256 hashed before storage; consumed and invalidated on first use
QR codeotpauth:// URI rendered to PNG via qrcode library — returned as data URL for authenticator app scanning
Login flowIf totpEnabled=true: credentials verified → TOTP token required before JWT issued

Role-Based Access Control (RBAC)

Permissions are strings (e.g. students.view, fees.collect, exams.enter-results) stored in the Role's permissions JSON column and embedded in every JWT. The RolesGuard reads the @RequirePermissions() decorator metadata via NestJS Reflector and checks that user.permissions contains every required permission.

// Controller decoration example @RequirePermissions('fees.collect', 'students.view') @Post(':studentId/collect') collectFee(@Param('studentId') id: string, @Body() dto: CollectFeeDto) { ... } // Guard implementation — all listed permissions must be present const required = reflector.getAllAndOverride<string[]>(PERMISSIONS_KEY, [ctx.getHandler(), ctx.getClass()]); if (!required.every(p => user.permissions.includes(p))) throw new ForbiddenException('Insufficient permissions');

Security Hardening

🔐
Cookie Security
All refresh tokens transmitted as HttpOnly, Secure (production), SameSite=Lax cookies — inaccessible to JavaScript, protected against CSRF. Separate cookie names per auth layer prevent cross-layer confusion.
✅
Input Validation
Global ValidationPipe with whitelist:true strips undeclared properties. forbidNonWhitelisted:true returns 400 on unknown fields. transform:true auto-coerces types. All DTOs use class-validator decorators.
🛡️
CORS Policy
Origin whitelist from CORS_ORIGIN env var (comma-separated). Requests with no Origin (server-to-server) are allowed. Unknown origins receive no CORS headers — browser blocks the request. credentials:true for cookie transmission.
🔒
Audit Logging
Global AuditLogInterceptor captures every POST/PUT/PATCH/DELETE. 9 sensitive fields auto-redacted: accessToken, refreshToken, token, passwordHash, password, newPassword, secret, tokenHash, generatedPassword. Written via setImmediate (zero latency impact).

External Integrations

Four external systems integrated — three mandatory (Cloudinary, Resend, Supabase) and one optional (N8N). All credentials are environment variables; no credentials in source code.

☁️
Cloudinary — File Storage
Pattern: Backend generates a signed upload URL (via Cloudinary SDK); frontend uploads the file directly to Cloudinary without the file ever passing through the NestJS server. This eliminates upload bottlenecks and multipart parsing overhead.

Used for: Student photos, institution logo, principal stamp, principal signature, student documents (Aadhaar, TC, birth certificate, etc.).

Storage path: <cloudName>/<institutionId>/<studentId>/
Deletion: publicId stored in StudentDocument — used to call Cloudinary destroy API on document removal.
📧
Resend — Transactional Email
Trigger points:
1. Forgot-password flow — 6-digit OTP emailed to staff/operator
2. New user account creation — generated password delivery (password immediately hashed; plain text not retained)
3. New parent account — portal credentials delivery

Template: HTML email via Resend API. The generatedPassword field is listed in the SENSITIVE_KEYS set and auto-redacted from all audit logs before any API response containing it is persisted.
🐘
Supabase + PgBouncer — Database
Dual URL pattern:
DATABASE_URL → port 6543 (PgBouncer transaction mode) — used for all runtime Prisma queries. Supports connection pooling for serverless bursts.

DIRECT_URL → port 5432 (direct PostgreSQL session mode) — used only for prisma migrate and prisma db push. Required because PgBouncer transaction mode does not support DDL statements.

Region: Singapore (ap-southeast-1) — collocated with Render backend.
⚙️
N8N — Optional Webhook Automation
Purpose: Event-driven workflow automation — optional integration point for WhatsApp notifications, fee reminders, admission confirmations, and report scheduling.

Integration pattern: Backend emits webhook POST to N8N endpoint on configured events. N8N handles downstream channel routing (WhatsApp Business API, SMS gateway, email sequences).

Configuration: Enabled via feature flag in Institution.features JSON — institutions on higher plans can activate automation workflows.

PlatformAdmin.activeSessionId is set on each login. The platform JWT embeds this sessionId. Every protected platform request validates that the JWT's sessionId matches the DB record. Section 16

Architecture Decisions

Key technical decisions with the reasoning behind each. These choices reflect deliberate trade-offs optimized for India's SME school market: fast time-to-market, low operational cost, and gradual scaling.

DecisionChoice MadeAlternatives ConsideredReasoning
Messaging model HTTP polling — client fetches messages at regular intervals WebSocket (socket.io), Server-Sent Events Polling eliminates WebSocket state management on the server, works behind all CDNs and load balancers without sticky sessions, and is trivially compatible with serverless/Render free-tier restarts. Acceptable latency for asynchronous parent-teacher communication.
Application structure NestJS modular monolith — all 21 modules in one process Microservices, separate fee/auth services Single deployment unit = zero inter-service network calls, simpler local development, no service mesh complexity. Module boundaries are clean enough to extract later if needed. Operational overhead of microservices not justified at current scale.
API paradigm REST with JSON bodies GraphQL, tRPC REST maps naturally to NestJS controller semantics, produces predictable audit log action strings (METHOD + URL), and is universally understood by frontend developers. GraphQL would require N+1 query protection and adds complexity for a data model that is mostly CRUD.
Multi-tenancy model Shared database, row-level isolation via institutionId Schema-per-tenant, database-per-tenant Single schema: zero migration coordination across tenants, instant provisioning (just INSERT a row), and standard Prisma tooling. FK cascades guarantee orphan-free deletes. Schema-per-tenant is the long-term migration path when isolation requirements grow.
Fee engine versioning V1 and V2 coexist in the same schema Big-bang replacement, feature flags only V1 data is live production data from existing schools — cannot be migrated without business-day disruption. V2 is additive (new models, no V1 columns dropped). Schools adopt V2 when convenient; both engines are query-compatible via the fee summary service layer.
Caching layer In-process AppCacheService (Map) in NestJS Redis, Memcached Eliminates external dependency and connection overhead. Sufficient for current single-instance deployment. Cache is not shared across instances (acceptable: most frequently cached data — institution profile, academic year — changes rarely and fetching fresh is safe). Redis migration path is straightforward when horizontal scaling is needed.
TOTP implementation Inline pure-Node implementation (zero extra packages) speakeasy, otpauth npm packages Zero additional attack surface, no transitive dependency risk. RFC 4226 + RFC 6238 implementation is ~60 lines of clear, auditable code. Only standard Node.js crypto primitives (HMAC-SHA1, randomBytes) used.
Audit log delivery setImmediate() fire-and-forget in NestJS interceptor Message queue (Bull, RabbitMQ), synchronous DB write Synchronous writes add latency to every mutating API call. A full message queue adds infrastructure complexity. setImmediate defers until after the HTTP response is flushed — adds zero p99 latency while still writing in the same process. Acceptable: audit log loss on crash is a known trade-off; crash recovery is rare.
Section 17

Future Roadmap

The platform is architected for incremental evolution. The following items represent planned capability expansions — each builds on the existing architecture without requiring disruptive rewrites.

🗃️
Schema-per-Tenant Isolation
Migration from shared-table row-level isolation to PostgreSQL schema-per-tenant. Each institution gets its own schema namespace (inst_abc123.students). Enables stronger regulatory isolation, schema-level backup, and enterprise compliance requirements. Prisma supports this via dynamic schema switching.
⚡
WebSocket Real-Time Messaging
Replace HTTP polling with WebSocket connections for the messaging module. NestJS Gateway (socket.io) with Redis adapter for multi-instance pub/sub. Enables instant message delivery, live typing indicators, and online presence. Polling remains as fallback for clients behind restrictive proxies.
🔴
Redis Shared Cache
Replace in-process Map cache with Redis for cross-instance cache sharing. Enables horizontal scaling without cache cold starts on additional instances. AppCacheService interface is already abstracted — Redis adapter is a drop-in replacement. Pub/sub channel also enables WebSocket multi-instance messaging.
📄
Server-Side PDF Generation
On-demand PDF generation for fee receipts, salary slips, report cards, and transfer certificates. Planned stack: Puppeteer or @react-pdf/renderer on the NestJS side. PDFs stored to Cloudinary and URL returned to client — avoids re-generating on repeat access. Critical for offline and audit scenarios.
🔔
Push Notifications
Firebase Cloud Messaging (FCM) integration for mobile push notifications. Trigger points: new message received, fee payment reminder, exam result published, announcement posted. Device token stored per user, sent via N8N webhook pipeline or direct FCM API from NestJS.
💸
Fee Engine V2 Full Migration
Complete migration of all institutions from V1 (FeeHead/FeeStructure/FeePayment) to V2 (FeePlan/FeeCollection). V1 tables deprecated after data migration scripts validate equivalence. Eliminates dual-engine complexity in the fees service layer and unifies the collection and reporting codebase.
📱
Native Mobile Application
React Native app targeting parents and students. Key flows: fee payment, attendance view, exam results, announcements, and messaging. The existing REST API is already designed for mobile consumption — no GraphQL or BFF layer needed. Zustand stores portable to React Native with minimal adaptation.
📊
Analytics & Reporting Dashboard
Cross-institution analytics for platform admin — enrollment trends, fee collection rates, attendance patterns, exam performance benchmarks. Institution-level dashboards for principal view. Powered by PostgreSQL analytical queries with optional materialized views for performance on large datasets.
