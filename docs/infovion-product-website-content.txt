SECTION 1 — WHAT IS THIS PRODUCT
================================================================

Product Name: Infovion Academic ERP
Tagline Options:
  - "One platform. Every role. Every school operation."
  - "School management, reimagined for India."
  - "From admission to exam results — managed in one place."

What it is:
Infovion Academic ERP is a cloud-based, multi-tenant school management
platform built specifically for K–12 schools in India. It replaces
scattered spreadsheets, WhatsApp chains, and disconnected software with
a single, role-aware system where every stakeholder — Director, Operator,
Principal, Teacher, Student, Parent, Accountant, Receptionist — gets
their own purpose-built portal.

It is not a generic SaaS template. It is built ground-up for the Indian
school system: Aadhar fields, Indian fee head standards, subject naming
conventions, TC (Transfer Certificate) workflows, and roll number systems
are baked in from day one.


================================================================
SECTION 2 — THE PROBLEM IT SOLVES
================================================================

Every small-to-mid school in India faces the same problems:

1. ADMISSION CHAOS
   - Inquiries tracked on paper or WhatsApp
   - No admission number system
   - TC verification done manually
   - No digital record of demographic info (Nationality, Caste, Religion,
     Aadhar) required by Indian school regulations

2. ATTENDANCE HEADACHES
   - Teachers mark attendance on registers
   - No way to alert parents when child is absent
   - No defaulter list for the principal
   - Month-end reports assembled manually

3. EXAMINATION DISORDER
   - Marks entered in Excel, shared over email
   - No unified scorecard or rank list
   - Admit cards printed manually
   - Results not accessible to students/parents

4. FEE CONFUSION
   - Fee collection tracked in ledgers
   - No balance view per student
   - No daily collection report for the accountant
   - No defaulter list for follow-up

5. ZERO PARENT VISIBILITY
   - Parents call the school for everything
   - No portal to check marks, fees, or attendance

6. STAFF MANAGEMENT GAPS
   - No staff attendance system
   - No leave request workflow
   - No staff directory with profiles

Infovion solves all of these — in a single login, role-specific interface,
accessible from any device, anywhere.

SECTION 5 — THE 8 PORTALS (ROLE-BASED UX)
================================================================

Every person in a school gets a portal. Not a single admin panel with
different buttons hidden. A completely separate, purpose-designed interface.

--- 1. DIRECTOR PORTAL ---
Who: School owner / Trustee / Managing Director
Route: /portal/director
What they can do:
  - Full school overview: student count, staff count, fee collection,
    attendance summary
  - Institution settings management
  - All module access (attendance, exams, fees, announcements, timetable)
  - Staff management: view all roles, reset passwords, deactivate accounts
  - Reports: attendance, fees, academic performance
  - Academic year and class structure management
  - Announcements: create school-wide and class-specific notices

--- 2. OPERATOR PORTAL (Admin Dashboard) ---
Who: School administrator / Office manager
Route: /dashboard
What they can do:
  - Student admission: full multi-step admission form
  - Student directory: class-first view, search, Excel export
  - Staff CRUD: create accounts, assign roles
  - Fee management: record payments, view balances, daily collection
  - Inquiry tracking: capture, follow up, convert to admission
  - Examination management: create exams, manage subjects
  - Attendance reports
  - Announcements

--- 3. PRINCIPAL PORTAL ---
Who: Academic head / Vice Principal
Route: /portal/principal
What they can do:
  - Overview: total students, academic year, class count, staff count
  - Attendance defaulters across all classes (configurable threshold)
  - Fee defaulters list
  - Attendance reports per class
  - Fee reports
  - Timetable view
  - Staff attendance tracking
  - Announcements management

--- 4. TEACHER PORTAL ---
Who: Class teachers, subject teachers
Route: /portal/teacher
What they can do:
  - Dashboard showing assigned classes
  - Attendance marking: select class → select date → mark each student
    (Present/Absent/Late/Leave), bulk "All Present" / "All Absent"
    Pre-fills if already marked for the day
  - Marks entry: select exam → select class → select subject →
    enter marks per student, mark absent, bulk save
  - Class timetable view
  - Announcements view
  - Staff attendance (own clock-in / clock-out)

--- 5. STUDENT PORTAL ---
Who: Enrolled students
Route: /portal/student
What they can do:
  - Own attendance history (monthly view)
  - Own exam marks and scorecard
  - Fee balance and payment history
  - Announcements / Notice board
  - Timetable view

--- 6. PARENT PORTAL ---
Who: Parent / Guardian of a student
Route: /portal/parent
What they can do:
  - Child's attendance (monthly view with absent alerts)
  - Child's exam results and scorecard
  - Child's fee balance and payment history
  - Announcements from school
  Real-time connection: parent account is linked to a specific student
  record — sees only their child's data

--- 7. RECEPTIONIST PORTAL ---
Who: Front desk staff, reception
Route: /portal/receptionist
What they can do:
  - Student inquiry capture
  - Fee payment recording
  - Announcements view
  - Student lookup / directory

--- 8. ACCOUNTANT PORTAL ---
Who: School accountant / finance staff
Route: /portal/accountant
What they can do:
  - Fee collection: daily collection report
  - Student fee balance view
  - Fee defaulter list (year-wise, class-wise)
  - Payment recording
  - Fee head and structure management

  - Framework: NestJS 11 (TypeScript)
  - Database: PostgreSQL (cloud-hosted on Neon, serverless)
  - ORM: Prisma
  - Authentication: JWT (passport-jwt) + httpOnly cookies

  - State management: Zustand with persistence

INFRASTRUCTURE
  - Database host: Neon (PostgreSQL, serverless, auto-scaling)
  - Backend host: Railway (Docker, auto-deploy from GitHub)
  - Frontend host: Vercel (global edge, auto-deploy from GitHub)
  - Repository: GitHub (BuildWithInfovion org)
  - CI/CD: GitHub → Railway + Vercel auto-deploy on push

WHY THIS STACK
  - Zero infrastructure management: Neon + Railway + Vercel handle scaling
  - India-accessible: works on 4G, mobile-first responsive design
  - No vendor lock-in: PostgreSQL is portable, NestJS is framework-agnostic
  - Cost-efficient for school budgets: cloud-native, pay-as-you-grow

SECTION 8 — ARCHITECTURE DECISIONS (TECHNICAL DEPTH FOR WEBSITE)
================================================================

MULTI-TENANCY APPROACH
  Design: Shared database with row-level isolation via institution_id.
  Every table that holds school data has an institution_id column.
  A TenantMiddleware reads the X-Institution-ID header on every API request
  and scopes all queries automatically.
  Future: Database-level Row-Level Security (RLS) policies planned for P2.

ROLE-BASED ACCESS CONTROL
  Roles and permissions stored in the database (not hardcoded).
  Every API endpoint is guarded by both AuthGuard (valid JWT) and
  RolesGuard (correct role).
  The JWT payload carries roles[] and permissions[] — no extra DB hit
  per request for authorization.

PORTAL ARCHITECTURE
  Each role gets a PortalShell component with its own nav, role badge,
  and guard. Navigating to the wrong portal auto-redirects to the correct one.
  Single source of truth: getRoleRoute() in auth-utils.ts.

TOKEN SECURITY
  Access token: 15-minute lifetime (short window limits exposure).
  Refresh token: 7-day rotation (new refresh token issued every time).
  Stored in httpOnly cookies (JavaScript cannot read them — XSS-safe).
  Refresh tokens stored hashed in DB. Revocation deletes the DB record.

AUDIT TRAIL
  Every write operation (create, update, delete) is logged with:
  user ID, institution ID, action, resource type, resource ID, timestamp.
  Non-deletable (append-only log table).

PERFORMANCE
  - In-memory TTL cache on hot read endpoints (class lists, subject lists)
  - DB indexes on institution_id, student_id, date columns
  - Slow query logging in production
  - Browser-level Cache-Control headers on static data responses
  - Batch DB operations in onboarding (createMany reduces 27 ops to ~9)
