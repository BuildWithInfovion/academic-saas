Remaining Bugs And Flaws

Critical

Frontend quality gates are effectively bypassed during production build

Evidence: apps/frontend/next.config.ts:3-9 sets typescript.ignoreBuildErrors = true and also includes an eslint build config key that Next.js 16 now reports as invalid. The build still succeeds while lint reports 125 errors.

Why this matters: shipping remains possible even when serious React, typing, and effect issues are present. That means regressions can go live undetected.

  • Reference: apps/frontend/next.config.ts
  • Command evidence: frontend build passed while warning about invalid config, and frontend lint failed with 148 problems.
Critical

Backend automated tests are currently broken and cannot protect core routes

Evidence: apps/backend/src/app.controller.ts:6-9 injects PrismaService, but apps/backend/src/app.controller.spec.ts:8-12 still registers only AppService. Running npm test fails immediately.

Why this matters: the backend has effectively lost its baseline regression safety net. Even small controller changes can now break routes without any test feedback.

  • Observed failure: Nest cannot resolve PrismaService in the test module.
  • This also shows the test suite is stale compared with the real application wiring.
High

React effect patterns are unstable across portal and dashboard pages

Evidence: lint reports repeated react-hooks/set-state-in-effect violations. Representative cases include apps/frontend/components/portal-shell.tsx:39-49, apps/frontend/app/portal/select-role/page.tsx:37-49, apps/frontend/app/portal/student/fees/page.tsx:41-57, and apps/frontend/app/dashboard/students/page.tsx:23-27.

Why this matters: these components synchronously set state inside effects or depend on unstable effect flows. That increases the chance of cascading renders, hydration flicker, stale data, and route-transition glitches in React 19 and newer lint rules.

  • The issue is not isolated to one screen; it appears in dashboard, principal, student, and role-selection flows.
  • Because auth and navigation shells are affected, these problems sit on critical user paths.
High

Frontend data layer is weakly typed, forcing unsafe casts throughout the UI

Evidence: apps/frontend/lib/api.ts:113-164 returns Promise<unknown>. As a result, many pages fall back to any casts and unchecked assumptions, for example apps/frontend/app/portal/student/fees/page.tsx:31-36 and apps/frontend/app/portal/student/fees/page.tsx:50-53.

Why this matters: the app builds, but type safety stops at the API boundary. Real API shape changes can silently break UI screens and only surface at runtime.

  • Lint evidence: many of the 125 frontend errors are @typescript-eslint/no-explicit-any.
  • This increases maintenance cost and slows down future feature work.
High

Backend request and middleware layers still rely heavily on unsafe untyped access

Evidence: backend lint shows broad no-unsafe-assignment, no-unsafe-member-access, and async misuse in key infrastructure files, including apps/backend/src/prisma/prisma.service.ts:63-117 and apps/backend/src/common/interceptors/audit-log.interceptor.ts:37-65.

Why this matters: guard, middleware, interceptor, and Prisma plumbing code sits below every module. Weak typing here makes security, tenancy, and audit behavior harder to trust and easier to break during refactors.

  • Lint also flags hard errors in infrastructure files, not just style warnings.
  • This is a cross-cutting platform risk rather than a single-feature bug.
Medium

Repo-level developer workflow is still incomplete

Evidence: the root package.json still defines "test": "echo \"Error: no test specified\" && exit 1". There is no working root validation command for the monorepo.

Why this matters: contributors cannot reliably run one standard command before pushing. That makes CI and local quality checks inconsistent and increases accidental breakage.

Medium

Platform onboarding still returns the generated operator password in the API response

Evidence: apps/backend/src/platform/platform.service.ts:539-546 returns operatorCredentials.password as plain text after client creation.

Why this matters: this may be intentional for first-time provisioning, but it increases exposure through browser devtools, network captures, support screenshots, and client-side logs unless the surrounding UI treats it very carefully.

Low

Minor code hygiene drift remains in backend modules

Evidence: backend lint still flags unused imports and unused locals in files like apps/backend/src/platform/platform.controller.ts:9, apps/backend/src/platform/platform.service.ts:4, and apps/backend/src/platform/platform.service.ts:438.

Why this matters: these are not production blockers by themselves, but they are a sign that refactors are leaving dead code behind and that the lint baseline is not actively enforced.

What Looks Good Right Now

  • The frontend production build completes successfully, so the current app can still be deployed.
  • The backend build also succeeds, which means the TypeScript server code is structurally compilable.
  • The recent navigation fixes around active portal and dashboard menu states appear integrated in the checked-in code.

Recommended Next Order Of Work

  1. Restore trust in validation: fix the backend test suite and stop ignoring frontend type errors in production builds.
  2. Stabilize auth and navigation surfaces first: clean up portal-shell, role selection, and student portal effect flows.
  3. Introduce typed API response helpers so pages stop falling back to any and unknown.
  4. Clean the backend infrastructure lint errors in guards, interceptors, and Prisma service before adding more modules.
  5. Add a real root workspace validation command so the repo can be checked consistently before every push.
