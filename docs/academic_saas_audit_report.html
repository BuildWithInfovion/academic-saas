Critical Issues

C1Cross-tenant data destruction via Subject module
subject.service.ts
removeSubjectFromUnit() deletes by academicUnitId + subjectId only; missing institutionId check enables cross-school deletion.
C2Hardcoded fallback JWT secret
auth.module.ts
Using || 'supersecret' silently signs tokens with a guessable key if JWT_SECRET is missing.
C3Timetable generator deletes without ownership check
timetable.service.ts
generate() can wipe any academic unit’s timetable if the unit ID is guessed.
C4Institution management endpoints unguarded
institution.controller.ts
Public create/list/seed endpoints allow anyone to create institutions and modify defaults.
C5Exam endpoints missing institution filter
exam.service.ts
Exam subject reads and writes can affect another school’s exam structure.
C6Academic module lacks permission guards
academic.controller.ts
Authenticated students and parents could create, edit, or delete academic structures.
C7Refresh token rotation not transactional
auth.service.ts
Two separate writes can leave both old and new refresh tokens valid after a crash.

High Issues

  • H1 — Audit log stores raw responses, including tokens and passwords.
  • H2 — No rate limiting on login, refresh, or forgot-password endpoints.
  • H3 — Public institution listing leaks school data.
  • H4 — Several queries are unbounded and can cause slowdowns or OOM.
  • H5 — Platform guard does not independently verify token expiry.
  • H6 — Missing cascade delete leaves orphaned rows.
  • H7 — Soft-delete collides with hard unique constraints.
  • H8 — No CSRF protection framework.
  • H9 — maxMarks can be altered after results exist.
  • H10 — Sequential fee receipt numbers are predictable.
  • H11 — Attendance save lacks institution ownership validation.

Medium Issues

  • M1 — Academic unit cycle creation is possible.
  • M2 — Attendance date comparison uses full DateTime instead of date-only logic.
  • M3 — Negative values are allowed for marks and fees.
  • M4 — Institution code normalization is inconsistent.
  • M5 — Student delete can orphan financial and exam records.
  • M6 — Staff attendance backdating lacks traceability.
  • M7 — Class teacher assignment lacks institution matching.
  • M8 — Password minimum length is too weak.
  • M9 — Student list UI hardcodes limit=200.
  • M10 — apiFetch retry flow may create redirect loops.
  • M11 — Announcement body needs sanitization safeguards.

Low Issues

  • L1 — Prisma conflict messages reveal field names.
  • L2 — /health is public; acceptable but should be documented.
  • L3 — Platform admin password policy is weak.
  • L4 — Audit timestamps are not UTC-normalized.
  • L5 — sessionStorage causes logout on new tab open.
  • L6 — No Content Security Policy headers are set.
  • L7 — Seed script contains hardcoded plaintext credentials.
  • L8 — Role changes lack before-state audit trail.
  • L9 — Inquiry module has no permission guards.
  • L10 — No database indexes documented for hot query columns.
