🔐 S1 — Missing Role-Based Access Control

Endpoints that lack @Permissions or @Roles decorators — any authenticated user from any role can call them. These are the most urgent findings.

S1-01 Attendance endpoints have NO permission guard — any logged-in user can mark any class absent Critical

All attendance endpoints use @UseGuards(AuthGuard, TenantGuard) but are completely missing RolesGuard or @Permissions(). A student or parent can log in and mark all students in any class as absent.

  • attendance.controller.ts — POST /attendance → Should require attendance.write (teacher/operator only)
  • attendance.controller.ts — GET /attendance/notifications/parent → Should require parent role only
  • attendance.controller.ts — GET /attendance/units/:unitId/defaulters → Should require operator/principal

Impact: Tampering with attendance records, falsified attendance reports, affects student progress and compliance.

S1-02 Exam endpoints have NO permission guard — any user can create exams and enter marks Critical

The entire exam module controller is unguarded at the permission level. A teacher can create exams (operator-only action). A student can post marks for themselves.

  • exam.controller.ts — POST /exams → Requires operator/principal
  • exam.controller.ts — POST /exams/results → Requires teacher (and only for their assigned subjects)
  • exam.controller.ts — PATCH /exams/:id/status → Requires operator
  • exam.controller.ts — POST /exams/:id/subjects → Requires operator

Impact: Students can self-modify marks. Teachers can publish exams they shouldn't control. Report cards become untrustworthy.

S1-03 Fee endpoints have NO permission guard — any user can record payments or delete fee structures Critical

The fees module has no role enforcement. Any authenticated user — including students and parents — can record a fee payment or delete a fee head.

  • fees.controller.ts — POST /fees/payments → Requires operator role
  • fees.controller.ts — POST /fees/heads → Requires operator
  • fees.controller.ts — DELETE /fees/heads/:id → Requires operator
  • fees.controller.ts — POST /fees/structures → Requires operator

Impact: Financial records can be fabricated or destroyed. Fee payments can be fraudulently recorded. High audit/compliance risk.

S1-04 Timetable module has NO role guard — any user can generate/overwrite any class's timetable Critical

Both read and write timetable endpoints are only protected by TenantGuard. Any authenticated user (student, parent) can overwrite the school timetable.

  • timetable.controller.ts — PUT /timetable/units/:unitId/slot
  • timetable.controller.ts — POST /timetable/units/:unitId/generate
  • timetable.controller.ts — GET /timetable/teacher/:teacherId → Any user can view any teacher's schedule
S1-05 Subject module write endpoints have no role check High

Any authenticated user can create, delete, and assign subjects to classes.

  • subject.controller.ts — POST /subjects
  • subject.controller.ts — DELETE /subjects/:id
  • subject.controller.ts — POST /subjects/units/:unitId
S1-06 Announcement create/update/delete has no author or role verification High

Any authenticated user can create school announcements, and any user can edit or delete any other user's announcement — there's no author ownership check.

  • announcement.controller.ts — POST /announcements → Should require operator/principal
  • announcement.service.ts — update() → No check that req.user.id === announcement.authorUserId
  • announcement.service.ts — delete() → Same gap — anyone can delete any announcement
S1-07 Academic unit delete endpoint missing @Permissions decorator High

academic.controller.ts — DELETE /academic/units/:unitId uses @UseGuards(RolesGuard) but no @Permissions('academic.write') decorator — the RolesGuard has nothing to check against, so it passes for any role.

Same issue on DELETE /academic/units/:unitId/class-teacher.

🏢 S2 — Multi-Tenancy Data Leaks

Queries or endpoints where institution scoping is missing or bypassable — could allow cross-school data access in a shared-schema multi-tenant system.

S2-01 DELETE /fees/structures/:id does NOT pass institutionId — any user can delete any institution's fee structure Critical

The controller does not extract institutionId from the request, and the service function deleteFeeStructure(id) only uses the raw record ID — no tenant scoping whatsoever.

// fees.controller.ts
@Delete('structures/:id')
deleteFeeStructure(@Param('id') id: string) {         // ← no @Request() req
  return this.feesService.deleteFeeStructure(id);     // ← no institutionId passed
}

Impact: Institution A can delete Institution B's fee structures if they know the UUID. Critical data loss risk in multi-tenant deployment.

S2-02 TenantGuard does NOT verify the institution exists or is active Critical

tenant.guard.ts reads the X-Institution-ID header and sets req.tenant = { institutionId } without checking that this institution actually exists in the database or is active/not suspended.

Anyone can send arbitrary UUID headers and reach protected endpoints scoped to a non-existent or suspended institution.

S2-03 Exam — removeExamSubject doesn't verify the subject belongs to the exam before deletion High

exam.service.ts — removeExamSubject() verifies the parent exam belongs to the institution, but then deletes the ExamSubject by ID alone using deleteMany({ where: { id, examId } }) without verifying that id actually belongs to this institution. A user can guess another institution's exam subject ID.

S2-04 Timetable controller uses req.institutionId directly — bypasses TenantGuard property High

Timetable controller accesses req.institutionId directly (a non-standard property) rather than using req.tenant?.institutionId (the property set by TenantGuard). If TenantGuard ever changes its property name, timetable data becomes unscoped.

S2-05 No DB-level Row Level Security — application middleware is the only tenant boundary Medium

The architecture relies entirely on application-level institution_id filtering. A single SQL bug in any service can expose all tenants' data. No PostgreSQL RLS policies are implemented (noted as P2 future work but worth flagging explicitly).

Impact: One missed where: { institutionId } in any service = full cross-tenant data dump.

🔑 S3 — Authentication Vulnerabilities

S3-01 Password reset approval returns the new password in plaintext in the API response Critical

auth.service.ts — approveResetRequest() generates a new password and returns { newPassword: '...' } in the HTTP response. This password travels in plaintext over the wire (even with HTTPS it appears in logs, proxy history, and browser DevTools).

The frontend Settings page app/dashboard/settings/page.tsx then displays this password in a modal — a user could screenshot or leave the tab open.

Fix: Return a time-limited reset token/link instead. The operator should communicate credentials via a separate channel (SMS/email).

S3-02 Password reset — no rate limiting per user/IP, only "one pending at a time" check High

auth.service.ts — requestPasswordReset() only blocks if a pending request already exists. Once approved (and the pending cleared), an attacker can flood the endpoint to lock out an account or spam the approval queue.

S3-03 Refresh token endpoint does not validate that the token belongs to the institution in the header High

auth.controller.ts — POST /auth/refresh takes institutionId from the tenant (header) and refreshToken from the body. If a token from Institution A is presented with Institution B's header, the refresh(institutionId, refreshToken) call will fail to find the token — but the error message may leak whether the token exists at all.

S3-04 LoginDto — password field missing @IsNotEmpty(), allows empty string login attempt Medium

auth/dto/login.dto.ts has @IsString() @MinLength(6) on password but no @IsNotEmpty(). An empty string fails MinLength but adds unnecessary noise to validation error messages instead of a clear "password is required" message.

S3-05 Platform routes bypass TenantMiddleware with no platform-admin guard in place Medium

tenant.middleware.ts skips all tenant validation for /platform routes. The platform controller handles multi-institution management (create institution, list clients) but no platform-admin JWT guard is verified to be enforced on every platform endpoint.

👁️ S4 — Sensitive Data Exposure

S4-01 Parent user created with phone only — no email, no password reset path via email High

student.service.ts — confirmAdmission() creates a parent user with only phone, no email. JWT payload email claim is empty. Password reset via email is impossible. Parent is permanently dependent on operator-mediated password reset.

S4-02 Student admission form collects Aadhar number, Caste, Religion — no data protection notice or access controls High

The student admission form collects PII sensitive under India's DPDP Act 2023 (Aadhar number, caste category, religion, nationality). No data access log, no field-level encryption, and any authenticated user with the student endpoint can read these fields. No UI-level data protection notice is shown to parents before submission.

S4-03 Staff attendance admin-override has no audit log Medium

staff-attendance.service.ts — adminMark() allows operators to alter any staff member's attendance for any date. markedById is stored, but no entry is written to the AuditLog table. There's no "change history" so an operator could retroactively alter attendance records invisibly.

S4-04 Password generated client-side in Director portal's staff management page Medium

app/portal/director/staff/page.tsx contains a generatePassword() function that generates passwords in the browser. The generated password is shown in the UI briefly. This is a pattern that can lead to weak passwords if the browser's Math.random() is not cryptographically secure — and it is not (Math.random is not CSPRNG).

⚙️ B1 — Business Logic Gaps

Backend rules that should be enforced but aren't — leading to invalid data states.

B1-01 Fee payment recorded without checking fee structure exists or calculating outstanding balance Critical

fees.service.ts — recordPayment() verifies student and fee head exist, then directly creates a payment record. It does NOT:

  • Check a FeeStructure exists for this student's class + year + fee head (can pay for fees not set up)
  • Check if the payment amount exceeds the outstanding balance (overpayment allowed)
  • Prevent duplicate payments for the same period
  • Validate that payment amount is in valid currency denomination

Impact: Accounting discrepancies, inability to reconcile fee ledgers, financial audit failures.

B1-02 Exam marks can be saved for students not belonging to the selected academic unit High

exam.service.ts — saveResults() validates that the ExamSubject entry (exam+unit+subject combination) exists, but does NOT verify that the student IDs in dto.entries actually belong to dto.academicUnitId. A teacher can post marks for students from other classes.

B1-03 Student "holdback" promotion action returns success with no actual database change High

student.service.ts — promote(): When action === 'holdback', the service returns { updated: students.length, action } immediately without updating any student record. The frontend shows a success message but nothing actually happened in the database. The student remains in the same class for the correct reason, but there's no record of the holdback decision.

B1-04 Attendance can be saved for any date — no validation for future dates or academic year bounds High

attendance.service.ts — save(): No date validation. Teachers can mark attendance for:

  • Dates in the future (pre-marking attendance)
  • Dates before the academic year's start date
  • Weekends or public holidays (no holiday calendar integration)
  • Dates from previous years (retroactive alteration)
B1-05 Staff leave request allows retroactive leave — no past-date check High

staff-attendance.service.ts — createLeaveRequest() only checks that endDate >= startDate. It allows creating leave requests for dates in the past, does not check for overlapping approved leaves, and has no holiday calendar awareness.

B1-06 Exam marks entry throws inside a transaction map — partial data saves can occur High

exam.service.ts — saveResults(): Validation (marksObtained > maxMarks) is done inside the dto.entries.map() function passed to prisma.$transaction(). If the exception fires mid-map, some upserts may have already been queued. The transaction may not roll back cleanly because the exception occurs before the Promise array is returned.

Fix: Validate all entries before calling $transaction().

B1-07 Class 1 TC exemption check is name-based (fragile) Medium

student.service.ts: The check for whether a student is in Class 1 (and therefore doesn't need a TC from a previous school) matches against unit names like ['class_1', 'class1', 'i', '1']. If a school names their Class 1 differently (e.g., "Grade 1", "Std I", "Class One"), TC validation will be incorrectly enforced.

B1-08 No duplicate admission prevention — student can be re-admitted after soft delete without detection Medium

Students are soft-deleted (deletedAt timestamp). A re-admission of the same student (same name + DOB + parent phone) will create a new record with a new admission number. The system has no deduplication check. Admission numbers will diverge from physical records.

🔌 B3 — API Design Issues

B3-01 No /students/me or /students/child endpoint guards — relies on student/parent linking not yet implemented High

The student portal pages call GET /students/me and the parent portal calls GET /students/child. These endpoints depend on a userId field on the Student model to link a login user to a student record — but this field is NOT YET in the Prisma schema. Both portals will always return the "not linked" state for every student and parent in the system.

Impact: Student and Parent portals are non-functional for all real users. Every student and parent login shows placeholder content only.

B3-02 Attendance defaulter query makes N+1 database calls — one per student in the class High

attendance.service.ts — getDefaulters() fetches the student list, then calls Promise.all(students.map(async (s) => {...})) — one async database query per student. For a class of 60 students, this is 61 DB queries per request. Principals viewing attendance across 5+ classes will generate 300+ queries per page load.

B3-03 Teacher sees ALL leaf academic units — not filtered to their assigned classes Medium

The attendance page and marks page for teachers call GET /academic/units/leaf which returns ALL leaf units across the institution. Teachers should only see units where they are assigned in academic_unit_subjects.teacherUserId. A teacher at this school can mark attendance for any class.

B3-04 No pagination on fee defaulters, attendance defaulters, exam results — potential large payload responses Medium

Several GET endpoints return unbounded lists: GET /fees/defaulters, GET /attendance/units/:id/defaulters, GET /exams/:id/results, GET /exams/:id/summary. A large school with 2000+ students will return massive payloads on every report page load.

B3-05 No soft-delete protection on fee heads — deleting a fee head with existing payments causes orphaned payment records Medium

fees.controller.ts — DELETE /fees/heads/:id: If a fee head is deleted but fee payments referencing it exist, the Prisma relation will either cascade-delete payment history (data loss) or throw a foreign key constraint error. Neither outcome is user-friendly and the behaviour depends on schema cascade rules not visible in the controller.

🚧 F3 — Placeholder / Stub Pages

Pages that exist in the navigation but show no functional content — visible dead ends for real users.

F3-01 Student Portal: all quick-action cards are purely decorative — no click handlers High

app/portal/student/page.tsx: The overview page shows feature preview cards (Attendance, Marks, Fees, Announcements) with no onClick or href. These do link via the sidebar navigation, but the cards themselves appear clickable without being so — confusing for students.

F3-02 Parent Portal: same decorative card issue — "Coming Soon" implied but not stated High

app/portal/parent/page.tsx: Mirrors the student portal issue. The child record not-linked banner + decorative cards create a confusing experience where a parent logs in and sees nothing actionable. No estimated timeline or "what to do next" guidance.

F3-03 Receptionist portal only has 2 menu items — Inquiries and Announcements Medium

app/portal/receptionist/layout.tsx: The Receptionist role is defined but its portal is severely limited. In most schools, the reception desk handles visitor logs, fee collection, outgoing TC requests, and appointment scheduling. None of these are accessible.

F3-04 Director Portal "Reports" page has no export/print — data visible but not extractable Medium

app/portal/director/reports/page.tsx: Shows attendance defaulters, fee defaulters, and exam list in tables. No export button, no print layout, no PDF download. A director presenting to management has no way to extract this data from the platform.

🎨 F4 — UI/UX Issues

F4-01 Settings page shows the new password in a modal — security-sensitive information on screen High

app/dashboard/settings/page.tsx: When an operator approves a password reset, the new auto-generated password is displayed in a modal. This password remains visible until the modal is dismissed. Any bystander can see it. No "copy to clipboard" with auto-clear, no hide/reveal toggle.

F4-02 Exam status cannot be changed from the frontend — backend supports draft/active/completed but no UI control High

app/dashboard/exams/page.tsx: Exam status is displayed as a badge but there are no buttons to transition status (draft → active → completed). The backend has PATCH /exams/:id/status but the frontend never calls it. Teachers will see "no active exams" even after the operator creates exams, because they're stuck in "draft".

F4-03 Staff list shows only email — no name field on User model, making it hard to identify staff Medium

The User model has no firstName / lastName fields. All staff pages display only email addresses. For large schools, finding "the Principal" vs "the Class 5A teacher" requires knowing their email address — highly impractical for operators and other staff.

F4-04 Teacher attendance marking: no visual distinction between "not yet marked" and "marked absent" Medium

app/portal/teacher/attendance/page.tsx: When loading an existing attendance session, status cycles P → A → L → Lv. But on initial load (no existing record), all students default to "Present". If a teacher accidentally saves without marking anyone — all 60 students are marked present by default with no warning.

F4-05 No confirmation dialog on destructive actions — deleting subjects, fee heads, academic units Medium

Delete buttons across the dashboard (subjects, fee heads, class structure) execute immediately with no "Are you sure?" confirmation. A mis-click on "Delete" for a fee head with 200+ payment records associated could cause data integrity issues.

F4-06 Academic year selector missing from many pages — implicit "current year" assumption with no indicator Medium

Pages like attendance marking, marks entry, and fee collection implicitly use the "current" academic year. No UI indicator shows which year is active. If the academic year is not set correctly (or if there are two years in the DB), all data goes to the wrong year silently.

F4-07 Hardcoded default max marks (100) and passing marks (35) in exam configuration UI Low

app/dashboard/exams/page.tsx: Default values of 100 and 35 are hardcoded in the exam configuration form. Schools with different marking schemes (e.g., 80-mark papers, 50-mark practicals, CBSE grading) must manually override every exam subject entry.

F4-08 Student directory and admission are two separate pages — confusing for operators Low

Sidebar has "Admission" (for creating new students) and "Student Directory" (for viewing/searching existing ones) as separate entries. There's no quick link from "Directory" to "New Admission". The student detail page at students/[id]/page.tsx links back to "Directory" but admission is a different page entirely.

🗄️ P2 — Data Model Gaps

P2-01 Student model has no userId field — student/parent portals cannot function Critical

The Prisma schema prisma/schema.prisma does not have a userId foreign key on the Student model. This means there is no way to associate a login user account with a student record. The spec §6.3 requires this field. Without it, the student and parent portals permanently show placeholder content.

P2-02 User model has no firstName/lastName — all staff identified by email only High

Every staff-related list, assignment view, and timetable displays email addresses instead of human names. This makes the platform difficult to use in practice. The schema migration needs: firstName String?, lastName String? on the User model.

P2-03 No holiday calendar model — attendance validation and leave calculation cannot reference school holidays High

The platform has no HolidayCalendar or SchoolHoliday table. Attendance can be marked on declared holidays, staff leave requests don't exclude holidays, and attendance percentage calculations include holiday days as "absent" — making all percentage calculations incorrect.

P2-04 No TransferCertificate model — TC workflow has no persistence layer High

Outgoing TCs (when a student leaves the school) are a legally mandated document in India. There is no TransferCertificate table, no TC number sequence, no TC request/approval workflow, and no issued-TC tracking. Currently, TC status is tracked only by a simple boolean flag on the student record.

P2-05 No blood group field in student admission form despite being in the student model Medium

The Student data model has a bloodGroup field (important for medical emergencies at school), but the admission form in app/dashboard/students/page.tsx does not include this field. Data mismatch between schema and UI means blood group is never captured at admission.

P2-06 No composite database index on (institutionId, email) for User table — slow login queries at scale Medium

Login queries filter by institutionId + email, but no composite index covers both fields together. At 1000+ users per institution, login latency will increase significantly. Schema change needed: @@index([institutionId, email]) on User model.

P2-07 Fee structure not linked to academic year — structures carry over year-to-year without reset Medium

When a new academic year is created, fee structures from the previous year are not automatically carried over or flagged for review. An operator must manually recreate all fee structures for the new year. Conversely, if they forget, the previous year's structures show up in the new year's payment forms.

🛠️ P3 — Operational / Process Gaps

P3-01 No first-login setup wizard — fresh institution has empty fee heads, subjects, and no academic year Critical

A new institution added via the platform admin must manually: create an academic year, call POST /institution/:id/seed-defaults to get default fee heads and subjects, set the current academic year. None of this is guided by the UI. If an operator starts adding students before setup is complete, they'll encounter broken dropdowns and missing fee structures with no explanation.

P3-02 No absent notification to parents when teacher marks student absent High

The backend has a GET /attendance/notifications/parent endpoint, suggesting a notification mechanism was planned. But there is no trigger from the attendance save flow to notify parents. Parent absent notifications (commonly via SMS or in-app) are a key engagement feature for school apps — and are completely absent.

P3-03 GitHub branch protection on main not configured — anyone with push access can force-push to production High

As noted in the project memory, GitHub branch protection rules on main and develop have not been set up. Without protection, any team member can push directly to main (the production branch) without a PR review.

P3-04 No rate limiting on any API endpoint — brute-force and DoS attacks possible Medium

No Throttler/rate-limit module is configured in the NestJS app. Login endpoint (POST /auth/login) can be brute-forced. Attendance defaulter queries (which generate N+1 queries) can be called in rapid succession to cause database load spikes.

P3-05 No error tracking / observability — production errors will be invisible Medium

No Sentry, Datadog, or equivalent error tracking is configured. When a production API call fails, the error is logged to the NestJS console (if a server is watching) but there's no alerting, no aggregation, and no ability to replay errors post-hoc.

P3-06 No CI/CD pipeline — deployments are manual, no automated test gate Medium

No GitHub Actions workflows exist. Every deployment is manual. No automated tests are run before a PR merge. A breaking change in the backend can reach production without any automated gate catching it.

P3-07 Student demographic fields (Nationality, Religion, Caste) buried in "Optional" collapsible Low

app/dashboard/students/page.tsx: Nationality, Religion, and Caste Category are inside an "Optional Details" collapsible section. Per Indian school compliance requirements (UDISE+ reporting), these are required fields — not optional. Operators will routinely skip them thinking they're optional, causing incomplete UDISE+ data.

P3-08 No Razorpay or online payment integration — all fees are offline/manual entry only Low

The platform is designed for India, where online fee payment via UPI/Razorpay is increasingly expected by parents. All fee payments are currently operator-entered (cash, cheque, NEFT). Parents cannot pay fees through the parent portal. This is listed as P1 future work but is a significant competitive gap.
